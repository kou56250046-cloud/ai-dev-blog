## はじめに

「いいプロンプトを書けばいい」と思っていた時代は、もう終わりつつある。

AI駆動開発の手法は、この1〜2年で驚くほど速く進化した。プロンプトエンジニアリングから始まり、コンテキストエンジニアリングを経て、現在は**ハーネスエンジニアリング**という新しいパラダイムへと移行しつつある。

それぞれが何を解決しようとして生まれたのか、そして今の自分の開発にどう取り入れるべきか——この記事でその全体像を整理する。

## ゴール

この記事を読むと以下ができるようになります：

- ✅ プロンプト・コンテキスト・ハーネスエンジニアリングの違いを説明できる
- ✅ なぜハーネスエンジニアリングが必要になったかを理解できる
- ✅ ハーネス設計の基本パターンを自分のプロジェクトに応用できる

**学ぶ意義**: AIを「ツール」として使う段階から「アーキテクチャとして設計する」段階に進むことで、品質・再現性・スケーラビリティが大きく向上する。

## 時間がない人のための要約

1. **プロンプトエンジニアリング** — 「何を言うか」で出力を制御する。限界は自己評価の甘さとコンテキスト消費。
2. **コンテキストエンジニアリング** — 「何を見せるか」で精度を上げる。情報の構造化と取捨選択が鍵。
3. **ハーネスエンジニアリング** — 「誰が何をするか」を役割分担する。生成・評価・修正を別エージェントが担うことで品質ループが回る。

## 手順

この記事は実装手順ではなく「概念の理解」がメイン。3つのフェーズを順番に追っていく。

---

### 1. プロンプトエンジニアリング — 「言い方を工夫する」時代

**何をしていたか**

指示文（プロンプト）を工夫することで、AIの出力品質を高める手法。
「ステップバイステップで考えて」「専門家として振る舞って」「JSON形式で出力して」——こういった書き方の工夫がすべてだった。

**何が問題だったか**

プロンプトだけでは限界がある。特に深刻なのが2点：

- **自己評価の甘さ**: AIは自分の出力を「完成した」と判断しがち。「この実装で問題ないか確認して」と聞いても、往々にして「問題ありません」と返ってくる。
- **コンテキスト消費問題（Context Anxiety）**: 長いタスクでコンテキスト上限が近づくと、AIは「未完成の機能を完成したと報告する」「UIの品質を落として早く終わらせようとする」という劣化挙動を示す。

プロンプトをどれだけ磨いても、**単一エージェントの構造的な限界**は回避できなかった。

---

### 2. コンテキストエンジニアリング — 「何を見せるか」を設計する

**何をしていたか**

AIに渡す情報（コンテキスト）を意図的に設計する手法。

- 不要なファイルは読ませない
- 必要な仕様書・型定義・エラーログを適切なタイミングで提供する
- `CLAUDE.md` のようなプロジェクト設定ファイルで常時参照情報を整備する
- RAG（Retrieval-Augmented Generation）でリアルタイムに関連情報を注入する

**何が変わったか**

「モデルの賢さ」に頼るのではなく、「モデルが見る情報の質」を管理することで、出力の精度と一貫性が大きく向上した。

**何がまだ問題か**

コンテキストを整備しても、**評価と生成が同一エージェントである**という問題は残る。自分で書いたコードを自分でレビューしても、見落としは減らない。これは人間でも同じだ。

---

### 3. ハーネスエンジニアリング — 「役割を分ける」構造設計

**何が変わったか**

ハーネスエンジニアリングでは、AIエージェントを**単一の万能担当者**として扱うのをやめる。

GAN（生成的敵対ネットワーク）のアーキテクチャにヒントを得た発想で、「生成する者」と「評価する者」を分離する。

```
[企画エージェント] → タスク分解・仕様策定
        ↓
[生成エージェント] → コード・コンテンツ作成
        ↓
[評価エージェント] → 品質チェック・問題点の指摘
        ↓
[修正エージェント] → フィードバックを受けて改善
        ↓
  ループ or 完了判定
```

**なぜ「別エージェント」が必要か**

同じエージェントが生成と評価を行うと、評価プロンプトをどれだけ工夫しても「自分の出力への甘さ」は消えない。

しかし**評価専用のエージェント**に「懐疑的な視点で評価せよ」「欠陥を探すことが役割だ」と明示的に指示すると、同じ指示を生成エージェント自身に与えるより大幅に機能する。

> **ポイント**: これは「AIが嘘をつく」問題ではなく、「役割バイアス」の問題。評価者と生成者を分けることで、バイアスを構造的に排除できる。

---

### 4. ハーネスエンジニアリングの実践指針

**① スプリント契約を事前に結ぶ**

実装を始める前に「何ができたら完了か」を明確に定義する。

```markdown
## 完了定義
- [ ] 指定された全コンポーネントが実装されている
- [ ] デザインシステムのカラー変数が正しく使われている
- [ ] モバイル表示が崩れていない
- [ ] TypeScriptの型エラーがゼロ
```

これがないと、評価エージェントが「何を基準に評価するか」が曖昧になる。

**② 評価基準を定量化する**

「良いかどうか」ではなく「何点か・何が足りないか」で評価させる。

```markdown
## 評価指示（評価エージェント向け）
以下の4軸で0〜10点で評価し、減点理由を具体的に列挙せよ：
1. デザイン品質（仕様書のカラーパレット・フォントへの準拠）
2. 独自性（テンプレートのコピーではなく設計意図が見えるか）
3. 実装品質（型安全・コンポーネント分離・再利用性）
4. 機能完成度（完了定義のチェックリストへの充足率）
```

**③ ループを前提に設計する**

一発で完成することを期待しない。生成→評価→修正のサイクルを繰り返すことを前提に、各エージェントの入出力インターフェースを設計する。

**④ ハーネスは減らせる**

モデルの能力が向上すれば、不要になるハーネスコンポーネントが出てくる。「ハーネスの組み合わせはシフトするが、消えるわけではない」——必要に応じて追加・削除できる柔軟な構造にしておく。

## 用語解説

| 用語 | 意味 |
|---|---|
| プロンプトエンジニアリング | AIへの指示文（プロンプト）の書き方を工夫してAIの出力を改善する手法 |
| コンテキストエンジニアリング | AIに渡す情報の内容・タイミング・構造を設計して精度を高める手法 |
| ハーネスエンジニアリング | 複数の専門エージェントを組み合わせ、生成・評価・修正の役割を分けるシステム設計 |
| Context Anxiety | コンテキスト上限に近づいたAIが「早く終わらせよう」とする劣化挙動 |
| GAN | 生成的敵対ネットワーク。「生成者」と「識別者」が競い合うことで品質を高めるニューラルネット手法 |
| スプリント契約 | 実装開始前に「何ができたら完了か」を明文化した合意書 |
| RAG | Retrieval-Augmented Generation。外部情報をリアルタイムにAIに注入して回答精度を高める手法 |

## まとめ

プロンプト→コンテキスト→ハーネスという進化は、「AIをどう扱うか」という問いへの答えが段階的に深まってきた歴史だ。

- プロンプトエンジニアリング：**言葉を磨く**
- コンテキストエンジニアリング：**情報を設計する**
- ハーネスエンジニアリング：**構造を設計する**

今すぐできる第一歩は、「生成と評価を同じエージェントにやらせない」というルールを1つのプロジェクトで試してみること。Claude Codeなら、コードを書くサブエージェントとレビューするサブエージェントを明示的に分けるだけで、品質の変化を体感できる。
