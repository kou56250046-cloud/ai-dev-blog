## はじめに

Claude Codeはファイルを書き換え、コマンドを実行し、GitHubにpushまで行える強力なツールだ。だからこそ、**一歩間違えると取り返しのつかないオペレーションを実行してしまうリスク**がある。

「AIが勝手にファイルを消した」「意図しないAPIキーがコードに埋め込まれた」「チームの誰かが全部許可で動かしていた」——これらは対策なしで使えば十分に起こりうる。

この記事では、Claude Codeを安全に使うための**7つの具体的な対策**を、「なぜそれが必要か」という理由とともに整理する。

## ゴール

この記事を読むと以下ができるようになります：

- ✅ Sandboxでの動作範囲の限定ができる
- ✅ denyルールで危険なコマンドの自律実行を防げる
- ✅ 機密ファイルへのアクセスをブロックできる
- ✅ PreToolUse Hookで実行前の自動チェックを仕込める
- ✅ チーム全体に統一したセキュリティポリシーを適用できる

**学ぶ意義**: 正しい設定で使えば、安心して自律エージェントに委任できる範囲が大きく広がる。セキュリティと自律性はトレードオフではない。

## 時間がない人のための要約

1. **Sandbox を有効化する** — 動作範囲をプロジェクト内に限定し、意図しないシステム破壊を防ぐ
2. **deny ルールを書く** — `rm -rf` / `git push -f` など取り返しのつかない操作を明示的に禁止する
3. **Hooks で事前チェックを自動化する** — 実行前に別エージェントが危険かどうかを判定する

## 前提条件

| 項目 | 条件 |
|---|---|
| Claude Code | インストール済み・基本操作ができる |
| OS | macOS / Linux / Windows（WSL推奨） |
| 対象者 | 個人開発〜チーム開発でClaude Codeを使っている人 |

## 1. サンドボックスの有効化

### なぜ必要か

デフォルト状態では、Claude Codeのファイルシステムとネットワークへのアクセス範囲が広い。**Sandboxを有効にすることで、動作範囲をプロジェクトディレクトリ内に限定**し、親ディレクトリやシステムファイルへの意図しない操作を防げる。

外部Webサービスと連携するタスクや、信頼度の低いスクリプトを実行するときに特に重要だ。

### 設定方法

```bash
# セッション内でSandboxを有効化
/sandbox
```

Sandbox内ではファイルシステムとネットワークが分離される。プロジェクト外への書き込みはデフォルトでもブロックされているが、Sandboxはネットワーク通信まで制限を加える。

```bash
# Sandboxあり：プロジェクト内への書き込みのみ許可
/my-project/src/components/Button.tsx  ✅

# Sandboxなし：親ディレクトリへの書き込みは許可確認が必要
/my-project/../other-project/secret.env  ❌（許可が必要）
```

> **ポイント**: 外部APIとの連携・Webスクレイピング・未知のスクリプト実行時は必ずSandboxを有効にする習慣をつける。

## 2. deny ルールで危険コマンドを禁止

### なぜ必要か

Claude Codeは自律的に判断してコマンドを提案・実行する。`rm -rf`・`git push --force`・`DROP TABLE`のような**取り返しのつかない操作を、AIが判断ミスして実行してしまうリスク**がある。

deny ルールで明示的に禁止しておくことで、どれだけ自律度を上げても最悪の事態を防ぐ安全装置になる。

### 設定方法（`.claude/settings.json`）

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(git reset --hard*)",
      "Bash(git clean -f*)",
      "Bash(sudo *)",
      "Bash(chmod 777*)",
      "Bash(kill -9*)"
    ]
  }
}
```

### CLAUDE.md に書く禁止事項

```
## 禁止コマンド（deny ルール）

以下は自律実行禁止。必ずユーザーの明示的な承認を得ること：

### ファイル・ディレクトリ操作
- rm -rf（再帰的削除）
- find ... -delete（条件付き一括削除）

### Git 操作
- git push --force / -f（強制プッシュ）
- git reset --hard（変更の強制破棄）
- git clean -f（未追跡ファイルの削除）
- git branch -D（ブランチ強制削除）

### データベース操作
- DROP TABLE / DROP DATABASE
- DELETE FROM（WHERE句なし）
- TRUNCATE

### システム操作
- sudo を伴うすべてのコマンド
- chmod 777
- kill -9
```

## 3. 機密ファイルへのアクセス制限

### なぜ必要か

`.env`ファイルや秘密鍵は、**一度内容が出力・提案・コードに埋め込まれると漏洩リスクが生まれる**。Claude Codeがこれらを読み込んで補完提案やログに含めてしまうケースを防ぐ必要がある。

特に`git diff`や`cat`の結果をそのままClaudeに渡すフローは危険だ。

### CLAUDE.md に書くアクセス制限

```
## 機密ファイルへのアクセス制限

### 読み取り禁止ファイル
- .env / .env.local / .env.production（環境変数）
- **/*.pem / **/*.key（秘密鍵・証明書）
- **/credentials.json（認証情報）
- ~/.ssh/**（SSHキー）

### 取り扱いルール
- 上記ファイルの内容を出力・提案・コードに埋め込まないこと
- APIキー・パスワードはコードに直書きせず、必ず環境変数参照にすること
- .gitignoreに機密ファイルが含まれているか確認してから作業すること
- シークレットの値をコミットメッセージに含めないこと
```

### 正しい環境変数の書き方

```bash
# ❌ 絶対にやってはいけない
ANTHROPIC_API_KEY=sk-ant-xxxx  # コードに直書き

# ✅ 正しい方法（環境変数参照）
ANTHROPIC_API_KEY="${ANTHROPIC_API_KEY}"
```

## 4. ネットワークのホワイトリスト設定

### なぜ必要か

Claude Codeが外部URLへリクエストを送る際、**プロンプトインジェクション経由で意図しないエンドポイントへデータが送信されるリスク**がある。接続を許可するドメインを明示的に限定することで、情報漏洩の経路を大幅に絞れる。

`curl` / `wget` はデフォルトでブロックされているが、明示的に許可した場合の運用ルールも定義しておく必要がある。

### CLAUDE.md に書くホワイトリスト

```
## ネットワークアクセスのホワイトリスト

### 許可するドメイン（明示的に許可されたもののみ）
- api.anthropic.com（Claude API）
- github.com / api.github.com（GitHub）
- [使用するDBのエンドポイント]
- [使用する外部APIのドメイン]

### 禁止する操作
- ホワイトリスト外ドメインへのリクエスト送信
- curl / wget による任意URLへのアクセス（明示的な許可なし）
- プロジェクトデータを外部サービスに送信する処理の自律実装

### 外部API連携時のルール
- 新しい外部APIを追加する場合はユーザーに確認してからホワイトリストに追加
- APIレスポンスをそのままClaudeに渡さない（プロンプトインジェクション対策）
```

## 5. PreToolUse Hook で通知

### なぜ必要か

Claude Codeが何かを実行する「直前」に通知を受け取ることで、**意図しない操作に気づけるようになる**。毎回手動で確認するのは現実的ではなく、Hookで自動化することで確認漏れをなくせる。

さらに、評価をClaude Code本体とは別のエージェントが行うことで、「自己評価の甘さ」問題も回避できる。

### 設定方法（`.claude/settings.json`）

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "以下のコマンドを安全性の観点で分析せよ。(1) 破壊的操作（rm -rf, dd, mkfs等） (2) 権限昇格（sudo） (3) ユーザー合意なしのネットワーク操作。'approve'または'deny'と理由を返せ。\nコマンド: $TOOL_INPUT.command",
            "timeout": 15
          }
        ]
      },
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "ファイル書き込みの安全性を確認せよ。システムパス・認証情報・パストラバーサル・機密内容が含まれていないか。'approve'または'deny'を返せ。"
          }
        ]
      }
    ],
    "ConfigChange": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '警告: Claude Code設定が変更されました。意図的な変更か確認してください。'"
          }
        ]
      }
    ]
  }
}
```

> **ポイント**: `ConfigChange` Hookを使えば、セキュリティ設定が変更されたときにもアラートを出せる。設定の無断変更を検知するための重要な仕組みだ。

## 6. /permissions で定期的な棚卸し

### なぜ必要か

allowlist（自動許可リスト）は**一度設定すると確認プロンプトが出なくなる**。開発が進むにつれて「あのとき追加したコマンド」が蓄積し、気づかないうちに危険な操作が自動許可されている状態になりやすい。

定期的な棚卸しでリストをクリーンに保つことが、長期的な安全運用の鍵だ。

### 棚卸し手順

```bash
# 現在のパーミッション設定を確認
/permissions
```

| タイミング | 棚卸しの内容 |
|---|---|
| 新フェーズ開始時 | 前フェーズで追加したallowlistを見直す |
| 外部ライブラリ追加時 | 新ツールに関連する許可設定を確認する |
| チームメンバー変更時 | 全メンバーの設定を統一し直す |
| 月1回の定期チェック | 使われていないエントリを削除する |

### allowlist に追加してよいコマンドの基準

```
✅ 追加OK（毎日使う・副作用なし）
- pnpm dev / pnpm build / pnpm test
- git status / git diff / git log
- astro check / astro sync

❌ 追加NG（破壊的・外部通信・副作用あり）
- rm 系コマンド
- git push（特に --force）
- curl / wget
- sudo を含むもの
```

## 7. 組織全体にポリシーを強制

### なぜ必要か

個人では安全に使えていても、**チームの誰か一人が「全部許可」の設定で動かしていれば組織全体のリスクになる**。Claude Codeのセキュリティは最も弱いメンバーの設定に引きずられる。

`.claude/settings.json` をGitで管理して全員に同じポリシーを適用することが、チーム運用の基本だ。

### 設定ファイルをGit管理する

```bash
# プロジェクトルートに配置してGit管理
.claude/settings.json
```

このファイルをリポジトリに含めることで、`git pull`するだけで全メンバーに同じセキュリティ設定が適用される。

### チーム向け settings.json のテンプレート

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(sudo *)",
      "Bash(DROP *)",
      "Bash(TRUNCATE *)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "コマンドの安全性を確認し、破壊的操作・権限昇格・未承認ネットワーク操作がないか判定。'approve'または'deny'を返せ。\nコマンド: $TOOL_INPUT.command",
            "timeout": 15
          }
        ]
      }
    ],
    "ConfigChange": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '警告: セキュリティ設定が変更されました'"
          }
        ]
      }
    ]
  }
}
```

### 新メンバーのオンボーディング手順

```
1. リポジトリをクローンする
2. .claude/settings.json が自動適用されていることを確認する（/permissions で確認）
3. このドキュメントのセキュリティルールを読む
4. 個人設定（~/.claude/settings.json）でチームポリシーを上書きしないこと
```

## 用語解説

| 用語 | 意味 |
|---|---|
| Sandbox | ファイルシステムとネットワークを隔離した安全な実行環境。`/sandbox` で有効化 |
| deny ルール | 特定のコマンドをClaude Codeが実行できないよう明示的に禁止する設定 |
| allowlist | 自動許可するコマンドのリスト。毎回確認プロンプトを出さずに実行できる |
| PreToolUse Hook | Claude Codeがツールを実行する直前に自動で走るスクリプト・AIプロンプト |
| ConfigChange Hook | Claude Codeの設定が変更されたときに発火するフック |
| /permissions | 現在のパーミッション設定を一覧表示するコマンド |
| managed settings | `.claude/settings.json` でチーム全体に適用するポリシー設定ファイル |
| プロンプトインジェクション | 外部コンテンツに悪意ある指示を埋め込み、AIに意図しない操作をさせる攻撃手法 |

## まとめ

Claude Codeのセキュリティ対策は「デフォルトで安全」を基本設計としているが、運用次第でリスクは生まれる。7つの対策を優先度順に整理すると：

| 優先度 | 対策 | 効果 |
|---|---|---|
| 🔴 最優先 | deny ルールの設定 | 取り返しのつかない操作を構造的に防ぐ |
| 🔴 最優先 | 機密ファイルのアクセス制限 | シークレット漏洩を防ぐ |
| 🟡 重要 | Sandbox の有効化 | 動作範囲を限定する |
| 🟡 重要 | PreToolUse Hook | 実行前の自動チェックを仕組み化する |
| 🟡 重要 | ネットワークホワイトリスト | 意図しない外部通信を防ぐ |
| 🟢 運用 | /permissions の定期棚卸し | 設定の肥大化・陳腐化を防ぐ |
| 🟢 運用 | 組織ポリシーの強制 | チーム全体の設定を統一する |

まず deny ルールと機密ファイル制限を設定して、次にHooksで自動化を加えていくのが実践的な導入ステップだ。
