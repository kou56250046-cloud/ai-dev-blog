## はじめに

「Claude Codeを使い始めたけど、なんとなく使っている気がする」

そう感じていませんか？Claude Codeは使い方次第で生産性に10倍近い差が出ます。その差を生むのが**ベストプラクティス**の有無です。

この記事では、ベストプラクティスの定義から、目的・用途別の具体的な実践パターンまでを体系的に整理します。公式ドキュメントを基に、実務で即使える形でまとめました。

## ゴール

この記事を読むと以下ができるようになります：

- ✅ ベストプラクティスの意義と定義を理解できる
- ✅ 新規開発・リファクタリング・デバッグ・テスト・チーム開発など用途別の使い方を実践できる

**学ぶ意義**: ベストプラクティスを知らずに使うと、同じ結果を得るのに3倍のトークン（＝コスト）と時間がかかる。正しいパターンを身につければ、Claude Codeは「高価なサジェスト補完」ではなく「自律するチームメンバー」になる。

## 時間がない人のための要約

1. **ベストプラクティスとは「再現性のある成功パターン」** — 試行錯誤の結果として蓄積された、誰が使っても同じ結果が出る手順
2. **最も重要なのは「CLAUDE.mdへの投資」** — 良い設計書がClaudeの指示精度を決める。コードより先に設計書を書く
3. **用途によって使い方が変わる** — 新規開発・デバッグ・リファクタリング・テストそれぞれに最適なパターンがある

## ベストプラクティスとは何か

### 定義

**ベストプラクティス（Best Practice）**とは、ある目的を達成するために「最も効果的と認められた手順・方法」のことです。個人の試行錯誤や組織の失敗から蓄積された、**再現性のある成功パターン**です。

「なんとなくうまくいった方法」ではなく、「なぜうまくいくのかが説明できる方法」がベストプラクティスです。

### Claude Codeにおけるベストプラクティスの重要性

```
ベストプラクティスなし：
  曖昧な指示 → Claudeが独断で判断 → 期待と違う結果 → 修正指示 → ループ

ベストプラクティスあり：
  明確な指示（設計書参照） → Claudeが意図を理解 → 期待通りの結果 → 完了
```

公式ドキュメントでは以下の3原則がベストプラクティスの根幹とされています：

1. **具体的な完了条件を定義する** — 「何ができたら終わりか」を先に書く
2. **CLAUDE.mdでプロジェクト規約を明文化する** — Claudeへの指示をファイルで永続化する
3. **小さな単位で反復する** — 大きなタスクはフェーズに分けて段階的に進める

## 手順：目的・用途別ベストプラクティス

### 【用途1】新規機能開発 — 完了条件を先に書く

新機能開発で最も重要なのは「何ができたら完成か」を最初に宣言することです。

**悪い例:**
```bash
claude -- "TODOアプリのAPIを作って"
# → 何をもって完成とするかが不明。Claudeが独断で判断する
```

**良い例（完了条件を明示）:**
```bash
claude -- "TODOリストのREST APIを作成して。

完了条件:
- GET/POST/PUT/DELETE のCRUDエンドポイントが動作すること
- 入力バリデーションが実装されていること
- テストカバレッジが80%以上であること
- README.mdにAPIドキュメントが書かれていること"
```

**フェーズに分けて段階的に進める:**

```bash
# Phase 1: 認証（完了後に次へ）
claude -- "Phase 1: JWTを使ったユーザー認証を実装して。
テストも書いて。完了したら教えて。"

# Phase 2: 商品一覧（Phase 1完了後）
claude -- "Phase 2: 商品カタログのリスト・検索機能を実装して。
Phase 1で作った認証ミドルウェアを活用すること。"

# Phase 3: カート機能
claude -- "Phase 3: ショッピングカートの追加・削除機能を実装して。"
```

> **ポイント**: 公式ドキュメントでは「大きなタスクをインクリメンタルなゴールに構造化すること」がベストプラクティスとして明記されています。

---

### 【用途2】デバッグ — 情報を先に渡す

デバッグで失敗するパターンは「エラーメッセージだけを渡す」ことです。Claudeはコンテキストが多いほど正確に診断できます。

**悪い例:**
```bash
claude -- "エラーが出てる。直して"
```

**良い例（情報を揃えてから渡す）:**
```bash
claude -- "以下のエラーを調査して修正して。

【エラーメッセージ】
TypeError: Cannot read properties of undefined (reading 'id')
  at getUserPosts (src/lib/posts.ts:42)

【発生条件】
新規登録したばかりのユーザーでログインした直後のみ発生

【直前に変更したファイル】
src/lib/auth.ts（セッション処理を修正）

【期待する動作】
ログイン直後でも記事一覧が表示されること"
```

**ブラウザエラーのデバッグ（Chrome連携）:**

```bash
# Chrome DevToolsと連携してライブデバッグ
claude -- "ブラウザでこのページを開いて、
コンソールエラーを確認し、原因のコードを修正して。
URL: http://localhost:3000/dashboard"
```

**自己修正サイクルを指示する（TDDパターン）:**

```bash
claude -- "認証機能をTDDで実装して。

手順:
1. 失敗するテストを書く
2. テストが通る最小限の実装をする
3. テストを実行する
4. 失敗したらデバッグ・修正する
5. リファクタリングする
6. すべてグリーンになるまで繰り返す"
```

---

### 【用途3】リファクタリング — 段階的・後方互換を守る

リファクタリングで最も大切なのは「動いているものを壊さない」ことです。

**基本パターン:**
```bash
# Step 1: 現状分析（まず把握させる）
claude -- "src/utils/ 以下のコードを分析して、
リファクタリングが必要な箇所をレポートして。
優先度（高/中/低）と理由も付けること。"

# Step 2: 小さな単位で実施（一気にやらない）
claude -- "分析レポートの優先度『高』の項目から始めて。
1ファイルずつ変更して、変更後に既存テストが通ることを確認すること。
後方互換性を必ず維持すること。"
```

**レガシーコードのモダン化:**
```bash
claude -- "src/api/ 以下のファイルをリファクタリングして。

変更内容:
- CommonJS (require) → ESModule (import) に移行
- コールバック → async/await に移行
- var → const/let に修正

条件:
- 1ファイルずつ変更する
- 変更後にテストが通ることを確認する
- 破壊的変更があれば変更前に報告すること"
```

> **公式の推奨**: 「リファクタリングは後方互換性を維持しながら小さな単位でテスト可能なインクリメントで行うこと」

### 【用途4】テスト自動化 — 「書いて・実行・直す」を一気に依頼する

テストの書き方で最も効果的なのは「書いて・実行・失敗を修正する」までを一度に依頼することです。

```bash
# テストを書いて実行・失敗も修正するまで完結させる
claude -- "auth モジュールのテストを書いて、実行して、
失敗があれば修正して。カバレッジ80%以上を目標にして"
```

**GitHub Actionsでテスト自動化:**

```yaml
# .github/workflows/claude-test.yml
name: Claude Code テスト自動化
on:
  schedule:
    - cron: "0 9 * * 1-5"  # 平日朝9時に自動実行
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "昨日のコミットで変更されたファイルのテストを実行して、
                   失敗があれば修正してPRを作成して"
```

**テスト種別ごとの指示パターン:**

```bash
# ユニットテスト
claude -- "src/utils/date.ts の全関数にユニットテストを書いて。
エッジケース（null・空文字・境界値）も必ずカバーすること"

# 統合テスト
claude -- "/api/posts エンドポイントの統合テストを書いて。
実際のDBを使うこと（モック禁止）。
正常系・異常系・認証エラーを含めること"

# E2Eテスト（Playwright連携）
claude -- "ユーザー登録から記事投稿までのE2Eテストを
Playwrightで書いて実行して"
```

### 【用途5】PR作成・コードレビュー — テンプレートを渡す

```bash
# PRレビューワークフロー
claude -- "PR #42 をレビューして。

確認項目:
1. コードの品質（可読性・保守性）
2. テストが存在するか
3. セキュリティ上の問題がないか（SQLインジェクション・XSS等）
4. CLAUDE.mdのコーディング規約に準拠しているか

問題を優先度（Critical/Minor）で分類してレポートして"
```

```bash
# PR自動作成（git push後に実行）
claude -- "現在の変更内容からPRを作成して。

PRの説明に含めること:
- 変更の背景と目的
- 変更した主なファイル一覧
- テスト結果のスクリーンショット（Playwright使用）
- レビュワーへの確認ポイント"
```

### 【用途6】チーム開発 — 並列セッションと知識の共有

**並列セッションで複数タスクを同時進行:**

```bash
# ターミナル1: バグ修正（独立したworktree）
claude --remote "認証のバグを修正して: src/auth/login.ts の42行目"

# ターミナル2: ドキュメント更新（同時並行）
claude --remote "APIドキュメントを最新の実装に合わせて更新して"

# ターミナル3: リファクタリング（独立した環境）
claude --remote "loggerをstructured outputに変更してリファクタして"
```

> **ポイント**: `--remote` フラグを使うと各タスクがクラウド上の独立した環境で実行されます。Gitのworktreeで分離されるため、タスク間でコードが干渉しません。

**チームのCLAUDE.mdを組織レベルで統一:**

```
組織レベルのCLAUDE.md配置場所:
  macOS: /Library/Application Support/ClaudeCode/CLAUDE.md
  Windows: %APPDATA%\ClaudeCode\CLAUDE.md

→ 全メンバーが同じルールでClaudeを使える
```

```bash
# リポジトリルートのCLAUDE.mdでチーム規約を共有
claude -- "CLAUDE.mdに以下のチーム規約を追記して：
- PRは機能単位で作成（1PR = 1機能）
- コミットメッセージは日本語OK
- テストなしのPRはマージ禁止
- Supabaseのマイグレーションは必ずレビューを通す"
```

### 【用途7】コンテキスト管理とコスト最適化

**CLAUDE.mdの肥大化を防ぐ:**

```markdown
<!-- 悪い例: CLAUDE.mdに詳細手順を全部書く（500行超え） -->
# デプロイ手順
1. テストを実行する
2. ビルドする
3. Vercelにpushする
... (200行続く)

<!-- 良い例: CLAUDE.mdは500行以内、詳細はSkillに分離 -->
# デプロイ手順
詳細は /deploy スキルを参照
```

**`/compact` でコンテキストを節約する:**

```bash
# 長いセッションの途中でコンテキストを整理
/compact

# → Claudeがこれまでの会話を自動要約し、コンテキストを解放する
# → 長時間セッションでも指示が無視されにくくなる
```

**用途に応じた処理強度の使い分け:**

| タスク | 推奨 | 理由 |
|---|---|---|
| 複雑なアーキテクチャ設計 | 高強度（デフォルト） | 深い推論が必要 |
| バグ修正・実装 | 中強度 | コストと品質のバランス |
| 単純なフォーマット・rename | 低強度 | トークン節約 |

## ベストプラクティス早見表

| 用途 | 最重要ポイント | アンチパターン |
|---|---|---|
| 新規開発 | 完了条件を先に宣言・フェーズ分割 | 「作って」だけ指示する |
| デバッグ | エラー・発生条件・変更履歴を一緒に渡す | エラー文だけ貼る |
| リファクタリング | 1ファイルずつ・後方互換確認 | 一気に全ファイル変更 |
| テスト | 「書いて・実行・修正」を一括依頼 | テストだけ書かせる |
| PRレビュー | 確認項目を箇条書きで渡す | 「レビューして」だけ |
| チーム開発 | `--remote` で並列・CLAUDE.mdで規約統一 | 全員バラバラに使う |
| コスト管理 | CLAUDE.md 500行以内・`/compact` 活用 | 設定なしで使い続ける |

## 用語解説

| 用語 | 意味 |
|---|---|
| ベストプラクティス | 試行錯誤の末に確立された、再現性のある最善手順 |
| コンテキスト | Claudeが会話中に保持している情報の総量。長くなると古い情報が失われる |
| `/compact` | コンテキストを要約して節約するClaude Codeのコマンド |
| Worktree | Gitの機能。同じリポジトリを複数の独立した作業ディレクトリで扱える仕組み |
| `--remote` | Claude Codeをクラウド上のサンドボックスで実行するフラグ |
| TDD | Test-Driven Development。テストを先に書いてから実装するアプローチ |
| インクリメンタル | 少しずつ段階的に進めること。一気に変更しないことでリスクを下げる |
| スキル（Skill） | CLAUDE.mdに書かず、呼び出したときだけロードされる専用の手順書 |

## まとめ

ベストプラクティスは「使いながら育てるもの」です。最初から完璧な設計書やCLAUDE.mdは存在しません。

**今すぐ始めるための3ステップ:**

1. **CLAUDE.mdを確認する** — 技術スタック・禁止事項・コマンドが書かれているか
2. **次の指示に「完了条件」を追加する** — 「作って」ではなく「〇〇ができたら完成」
3. **大きなタスクをPhaseに分ける** — Phase 1・2・3に分割してから依頼する

この3つだけでも、Claude Codeの出力品質は大幅に変わります。
