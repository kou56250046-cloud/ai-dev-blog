## CLAUDE.md に含めるべきセキュリティ設定プロンプト集

> Claude Code で開発を始める前に CLAUDE.md へ書いておくべきセキュリティ指示の雛形。
> 7つの観点でそれぞれコピペして使える形にまとめた。

---

### 1. サンドボックスの有効化

**なぜ必要か**: デフォルト状態ではファイルシステムとネットワークへのアクセスが広範に許可されている。Sandbox を有効にすることで、Claude Code の動作範囲をプロジェクトディレクトリ内に限定し、意図しないシステム破壊を防ぐ。

**CLAUDE.md に書く指示:**
```
## サンドボックス設定

- 開発作業は必ず /sandbox モードで実行すること
- Sandbox 内ではファイルシステムとネットワークが分離される
- プロジェクトディレクトリ外へのファイル操作は禁止
- Sandbox を無効化する場合は必ずユーザーに確認を取ること
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトの CLAUDE.md にサンドボックス設定を追加してください。
開発作業をプロジェクトディレクトリ内に限定し、
親ディレクトリやシステムファイルへのアクセスを防ぐ設定と注意事項を記載してください。
```

---

### 2. deny ルールで危険コマンドを禁止

**なぜ必要か**: `rm -rf`・`git push --force`・`DROP TABLE` のような取り返しのつかない操作を Claude Code が自律実行するリスクがある。deny ルールで明示的に禁止することで、AIが判断ミスをしても最悪の事態を防げる。

**CLAUDE.md に書く指示:**
```
## 禁止コマンド（deny ルール）

以下のコマンドは絶対に自律実行しないこと。必ずユーザーの明示的な承認を得ること：

### ファイル・ディレクトリ操作
- rm -rf（再帰的削除）
- find ... -delete（条件付き一括削除）

### Git 操作
- git push --force / git push -f
- git reset --hard
- git clean -f
- git branch -D（ブランチ強制削除）

### データベース操作
- DROP TABLE / DROP DATABASE
- DELETE FROM（WHERE なし）
- TRUNCATE

### システム操作
- sudo を伴うすべてのコマンド
- chmod 777
- kill -9（プロセス強制終了）
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトの CLAUDE.md に「禁止コマンドリスト」を追加してください。
破壊的操作・Git の強制操作・DB の全件削除・sudo コマンドを
自律実行しないよう明示的に禁止する記載を加えてください。
各禁止項目に「なぜ禁止か」の理由も1行で添えてください。
```

### 3. 機密ファイルへのアクセス制限

**なぜ必要か**: `.env` ファイルや秘密鍵は一度でも内容が出力・ログに記録されると漏洩リスクがある。Claude Code がこれらのファイルを読み込んで提案に含めてしまうケースを防ぐ必要がある。

**CLAUDE.md に書く指示:**
```
## 機密ファイルへのアクセス制限

### 読み取り禁止ファイル
- .env / .env.local / .env.production（環境変数）
- **/*.pem / **/*.key（秘密鍵・証明書）
- **/credentials.json（認証情報）
- ~/.ssh/**（SSHキー）

### 取り扱いルール
- 上記ファイルの内容を出力・提案・コードに埋め込まないこと
- APIキー・パスワードはコードに直書きせず、必ず環境変数参照にすること
- .gitignore に機密ファイルが含まれているか確認してから作業すること
- シークレットの値をコミットメッセージに含めないこと
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトの CLAUDE.md に機密ファイルへのアクセス制限を追加してください。
.env ファイル・秘密鍵・認証情報ファイルを読み取り・出力・コードへの埋め込みから
保護するルールを記載してください。
また .gitignore に漏れがないかチェックする手順も含めてください。
```

### 4. ネットワークのホワイトリスト設定

**なぜ必要か**: Claude Code が外部 URL へリクエストを送る際、意図しないエンドポイントへのデータ送信（プロンプトインジェクション経由の情報漏洩）が起きうる。接続を許可するドメインを明示的に限定することでリスクを下げる。

**CLAUDE.md に書く指示:**
```
## ネットワークアクセスのホワイトリスト

### 許可するドメイン（明示的に許可されたもののみ）
- api.anthropic.com（Claude API）
- github.com / api.github.com（GitHub）
- [使用するDBのエンドポイント]
- [使用する外部APIのドメイン]

### 禁止する操作
- ホワイトリスト外のドメインへのリクエスト送信
- curl / wget による任意 URL へのアクセス（明示的な許可なし）
- プロジェクトデータを外部サービスに送信する処理の自律実装

### 外部 API 連携時のルール
- 新しい外部 API を追加する場合はユーザーに確認してからホワイトリストに追加する
- レスポンスデータをそのまま Claude に渡さない（プロンプトインジェクション対策）
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトの CLAUDE.md にネットワークアクセスのホワイトリスト設定を追加してください。
許可するドメイン一覧・禁止する通信操作・外部 API 追加時のルールを記載してください。
プロンプトインジェクション対策の観点も含めてください。
```

### 5. PreToolUse Hook で通知

**なぜ必要か**: Claude Code が何かを実行する「直前」に通知を受け取ることで、意図しない操作に気づけるようになる。特に破壊的な操作・外部通信・設定変更の前に割り込む仕組みが重要。

**CLAUDE.md に書く指示:**
```
## PreToolUse Hook 設定

以下の操作の実行前に必ず通知・確認を行うこと：

### 通知が必要な操作
- Bash コマンドの実行（特に rm / git / curl を含むもの）
- ファイルの新規作成・上書き（Write / Edit）
- 設定ファイルの変更（.claude/ 以下・package.json 等）
- 外部ネットワークへのリクエスト

### hooks 設定例（.claude/settings.json）
PreToolUse に Bash / Write / Edit のマッチャーを設定し、
実行内容をログに記録・ユーザーに通知するスクリプトを組み込む。
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトに PreToolUse Hook を設定してください。
Bash コマンド実行・ファイル書き込み・設定変更の前に
通知が出るよう .claude/settings.json を作成または更新してください。
特に rm / git push / curl を含むコマンドは内容をログに残してください。
```

### 6. /permissions で定期的な棚卸し

**なぜ必要か**: allowlist（自動許可リスト）は一度設定すると確認プロンプトが出なくなるため、いつの間にか不要・危険な許可が蓄積しやすい。定期的に見直してリストをクリーンに保つことが重要。

**CLAUDE.md に書く指示:**
```
## パーミッション棚卸し運用ルール

### 棚卸しのタイミング
- 新機能の開発フェーズ開始時
- 外部ライブラリ・ツールを追加したとき
- チームメンバーが変わったとき
- 月1回の定期チェック

### 棚卸し手順
1. /permissions コマンドで現在の許可設定を確認する
2. 使われていない allowlist エントリを削除する
3. 本当に自動許可が必要なコマンドだけを残す
4. チームで設定を共有し .claude/settings.json を Git にコミットする

### allowlist 追加の基準
- 毎日使う安全なコマンドのみ（pnpm dev / git status 等）
- 破壊的操作・外部通信を含むコマンドは allowlist に追加しない
```

**Claude Code への依頼プロンプト:**
```
現在の /permissions 設定を確認して、以下を実施してください：
1. allowlist に登録されているコマンドを一覧表示する
2. 不要・危険なエントリを特定して削除を提案する
3. 残すべきエントリとその理由を説明する
4. 棚卸し後の設定を .claude/settings.json に反映する
```

### 7. 組織全体にポリシーを強制

**なぜ必要か**: 個人が安全に使えていても、チームの誰か一人が「全部許可」の設定で動かしていれば組織全体のリスクになる。managed settings で設定を Git 管理し、全員に同じセキュリティポリシーを適用することが重要。

**CLAUDE.md に書く指示:**
```
## 組織・チームへのポリシー適用

### 設定ファイルの管理
- .claude/settings.json を Git で管理し全メンバーに共有する
- 個人設定（~/.claude/settings.json）でチームポリシーを上書きしないこと
- セキュリティ設定の変更は PR レビューを経てからマージする

### 全員に適用するルール
- 本番 DB への直接接続は禁止（開発用 DB のみ使用）
- 本番環境の環境変数を含む .env.production は Claude Code に読ませない
- チームの allowlist は最小限に保ち、追加時はチームで合意する

### 新メンバーのオンボーディング
1. .claude/settings.json の内容を説明する
2. このセクションのルールを周知する
3. /permissions で設定が正しく読み込まれているか確認させる
```

**Claude Code への依頼プロンプト:**
```
このプロジェクトのチーム向けセキュリティポリシーを
.claude/settings.json として作成してください。
以下を含めてください：
- チーム全員に適用する deny ルール
- allowlist の初期設定（最小限）
- 本番環境アクセスの禁止設定
- ConfigChange Hook（設定変更時の通知）
Git で管理できる形式で出力してください。
```
